# Design: Remove the Airflow `ReadWriteMany` PVC

**Date:** 2026-04-23

## Goal

Remove the `dbt-target` `ReadWriteMany` PVC that Airflow executor pods currently mount at `/shared/dbt-target`. This is the only RWX PVC in the deployment, and it is the single reason the demo is pinned to AKS (`azurefile` storage class). Eliminating it unblocks running the demo on vanilla Kubernetes clusters without an RWX-capable storage class.

## Non-goals

- Changing the Cosmos per-model execution model (we keep `DbtTaskGroup` + `ExecutionMode.VIRTUALENV`; the per-model task visibility in the Airflow UI is preserved).
- Moving dbt out of an ephemeral per-pod virtualenv into the Airflow image.
- Refactoring OpenMetadata ingestion wiring. The S3 prefix OM reads from (`s3://dbt-artifacts/tpch_demo/`) stays the same.
- Any unrelated README / AKS-limitations cleanup beyond the RWX justification.

## Why the PVC exists today

1. Airflow uses `kubernetesExecutors` — every task runs in its own pod, potentially on a different node.
2. The dbt project lives in a read-only git-sync mount at `/stackable/app/git-0/current/dags/dbt/tpch_demo`, so dbt cannot write to the default `<project>/target/` path. `DBT_TARGET_PATH=/shared/dbt-target` redirects dbt's output to a writable location.
3. After per-model Cosmos tasks produce `run_results.json` (and share `manifest.json`) in that target dir, the subsequent `upload_dbt_artifacts` PythonOperator — running in a *different* executor pod — reads those files and pushes them to `s3://dbt-artifacts/tpch_demo/` for OpenMetadata ingestion.

Cross-pod sharing forces RWX. No other component uses RWX (logs go to S3 via `REMOTE_BASE_LOG_FOLDER`; DAGs use git-sync per pod).

## Approach

Replace the shared volume with an S3-mediated shard-and-merge pattern:

1. **Writable target dir stays pod-local.** `DBT_TARGET_PATH` is set to `/tmp/dbt-target` (tmpfs inside each executor pod). Dbt writes its default artifacts there.
2. **Per-task upload via callback.** Every Cosmos-generated dbt task gets an `on_success_callback` and `on_failure_callback` that uploads *only* its `run_results.json` to a per-run shard location in GarageFS. The full `manifest.json` does not need to be uploaded — the finalize task regenerates it.
3. **Finalize task does everything else.** A new `finalize_dbt_artifacts` PythonVirtualenvOperator replaces the existing `upload_dbt_artifacts`. It:
   1. Lists and downloads all run_results shards for this DAG run.
   2. Merges the shards into one canonical `run_results.json`.
   3. Runs `dbt docs generate`, which compiles the project (producing `manifest.json`) and queries Trino's information_schema (producing `catalog.json`) — both land in the finalize pod's target dir.
   4. Uploads the three canonical artifacts to `s3://dbt-artifacts/tpch_demo/`.
   5. Deletes the shard prefix for this run.
4. **OM trigger tasks are unchanged.** They still read from the same canonical S3 prefix.

### DAG shape

Before:

```
wait_for_services >> dbt_tasks >> upload_artifacts
                              >> trigger_metadata_ingestion
                              >> trigger_dbt_ingestion
```

After:

```
wait_for_services >> dbt_tasks >> finalize_dbt_artifacts
                              >> trigger_metadata_ingestion
                              >> trigger_dbt_ingestion
```

### S3 layout

- Canonical (read by OM, unchanged): `s3://dbt-artifacts/tpch_demo/{manifest,catalog,run_results}.json`
- Per-run shards (new, internal): `s3://dbt-artifacts/_runs/<url-encoded-dag_run_id>/run_results/<task_id>.json`

Shards are deleted by `finalize_dbt_artifacts` after a successful merge.

### Why regenerating manifest/catalog in finalize is safe

- `dbt docs generate` compiles the dbt project from source files and writes `manifest.json` byte-equivalent (modulo timestamps) to the manifests in per-model task pods. Node `unique_id`s match, so the merged `run_results.json` entries line up with the finalize-generated manifest.
- `catalog.json` is generated by querying Trino's `information_schema` for every relation. It reflects the post-run warehouse state, which is what OpenMetadata wants for column-level metadata.

### dbt environment for the finalize task

Use `PythonVirtualenvOperator` with `requirements=["dbt-trino"]` and `system_site_packages=True`. The venv inherits the Airflow image's site-packages (airflow, providers, boto3 — all already available to the current `upload_dbt_artifacts` task) and layers `dbt-trino` on top. This mirrors the spirit of the `ExecutionMode.VIRTUALENV` pattern Cosmos already uses for per-model tasks: dbt-trino stays confined to an ephemeral per-pod virtualenv and never enters the Airflow image. The scheduler, webserver, and triggerer pods remain unaffected. Cost: ~30–60 s pip cold start once per DAG run. `system_site_packages=True` is chosen (rather than `False`) specifically so the callable can use `airflow.hooks.base.BaseHook` to resolve the GarageFS connection without re-installing airflow inside the venv.

## Components

### Per-task callback

```python
def upload_run_results_shard(context):
    """Upload this task's per-model run_results.json to a per-run shard key.

    Runs as Airflow on_success_callback / on_failure_callback, inside the task
    executor pod, after dbt has exited. Failed tasks often still produce a valid
    run_results.json (e.g. test failures) that OM wants to ingest.
    """
    from pathlib import Path
    from urllib.parse import quote
    local = Path("/tmp/dbt-target/run_results.json")
    if not local.exists():
        return  # task died before dbt wrote anything
    run_id = quote(context["dag_run"].run_id, safe="")
    task_id = context["ti"].task_id
    key = f"_runs/{run_id}/run_results/{task_id}.json"
    _s3_client().upload_file(str(local), "dbt-artifacts", key)
```

Wired into the `DbtTaskGroup` via `default_args`:

```python
DbtTaskGroup(
    ...,
    default_args={
        "retries": 1,
        "on_success_callback": upload_run_results_shard,
        "on_failure_callback": upload_run_results_shard,
    },
)
```

Shard upload exceptions surface in the Airflow task log but do not override the task's dbt-result status. This is an accepted trade-off for this demo's data volume; we do not add callback-level retries.

### Finalize task

```python
def finalize_dbt_artifacts(**context):
    import json, os
    from pathlib import Path
    from urllib.parse import quote
    from dbt.cli.main import dbtRunner

    run_id = quote(context["dag_run"].run_id, safe="")
    s3 = _s3_client()

    shards = _list_shards(s3, f"_runs/{run_id}/run_results/")
    merged = _merge_run_results([_download_json(s3, k) for k in shards])
    local_target = Path("/tmp/dbt-target")
    local_target.mkdir(parents=True, exist_ok=True)
    (local_target / "run_results.json").write_text(json.dumps(merged))

    env = os.environ | {"DBT_TARGET_PATH": str(local_target)}
    result = dbtRunner().invoke(
        ["docs", "generate",
         "--project-dir", str(DBT_PROJECT_PATH),
         "--profiles-dir", str(DBT_PROJECT_PATH)],
        env=env,
    )
    if not result.success:
        raise RuntimeError(f"dbt docs generate failed: {result.exception}")

    for name in ("manifest.json", "catalog.json", "run_results.json"):
        s3.upload_file(str(local_target / name), "dbt-artifacts", f"tpch_demo/{name}")

    _delete_prefix(s3, f"_runs/{run_id}/")
```

Operator wiring:

```python
finalize = PythonVirtualenvOperator(
    task_id="finalize_dbt_artifacts",
    python_callable=finalize_dbt_artifacts,
    requirements=["dbt-trino"],
    system_site_packages=True,
    op_kwargs={
        "dbt_project_path": str(DBT_PROJECT_PATH),
        "dbt_target_path": "/tmp/dbt-target",
        "s3_bucket": "dbt-artifacts",
        "s3_prefix": "tpch_demo",
    },
)
```

**Implementation constraint for `PythonVirtualenvOperator`:** the callable is serialized and run in a subprocess, so it cannot reference module-level helpers or constants from the DAG file. All helpers (`_s3_client`, `_list_shards`, `_merge_run_results`, `_download_json`, `_delete_prefix`) must be defined inside `finalize_dbt_artifacts`, and all values must arrive via `op_kwargs`. The pseudocode above simplifies this for readability; the actual implementation inlines helpers.

### `run_results.json` merge semantics

Each shard is a full, valid `run_results.json` whose `results` array has exactly one entry (Cosmos invokes dbt with one node per task — either a `run` for a model or a `test` for a test; both produce the same envelope). Merge:

- Concatenate all `results` arrays into a single list.
- Use any shard's `metadata` and `args` as the envelope (they are effectively identical across shards).
- Set `elapsed_time` to the sum of per-entry `execution_time` values. (OM uses `execution_time` on each result, not the top-level `elapsed_time`, so this field is not load-bearing.)

### Failure modes

| Situation | Outcome |
|---|---|
| Per-model task crashes before dbt writes | No shard for that task. Merged run_results omits the model's result; manifest still lists the model. OM shows "not run this round" — accurate. |
| Concurrent DAG runs | Shards are scoped by `dag_run_id`, so runs do not collide. |
| Callback S3 upload fails | Logged on the task; task status is preserved. A follow-up run republishes. No additional retry logic. |
| `dbt docs generate` fails (Trino unreachable) | Finalize task fails. OM triggers do not run. Retry semantics inherit from `default_args`. |
| One or more dbt tasks fail | `finalize` is skipped by default trigger rule `all_success` — matches today's behavior, nothing worse. |

## Testing

**Pre-flight**
- `just dbt-compile` still works (no change to the dbt project).

**In-cluster golden path**
1. Deploy or update the cluster with the changed manifests.
2. Confirm `kubectl get pvc -n platform` no longer lists `dbt-target`.
3. Unpause / trigger the `dbt_tpch_demo` DAG.
4. Watch per-model executor pods run to completion without "read-only filesystem" errors when dbt writes to `/tmp/dbt-target`.
5. After `finalize_dbt_artifacts` succeeds, verify via `mc` / `aws` CLI that `s3://dbt-artifacts/tpch_demo/{manifest,catalog,run_results}.json` are present and non-empty, and that `s3://dbt-artifacts/_runs/` no longer has keys for that run.
6. Confirm `trigger_om_metadata_ingestion` and `trigger_om_dbt_ingestion` run to success; in the OpenMetadata UI, verify the Trino service's tables show dbt-linked models with descriptions and test nodes.

**Edge case — dbt task failure**
- Introduce a deliberate SQL error in one mart model. Run the DAG. Confirm: the failing task's shard is uploaded via `on_failure_callback`; `finalize` is skipped by the `all_success` trigger rule (matches current behavior); the DAG is marked failed.

**Acceptance — non-AKS cluster**
- Provision a vanilla Kubernetes cluster (kind, k3d, EKS, GKE) with no RWX storage class.
- Run `just deploy` through to a successful DAG run and OpenMetadata dbt ingestion.
- This is the true acceptance criterion: the demo runs end-to-end without azurefile.
