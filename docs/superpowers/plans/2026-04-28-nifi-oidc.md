# NiFi OIDC SSO — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace NiFi's static `simple-nifi-users` AuthenticationClass with a Keycloak OIDC AuthenticationClass. `demo-admin` is the initial admin who can edit policies inside NiFi; `demo-user` logs in as a non-privileged user.

**Architecture:** New `AuthenticationClass: nifi-keycloak` + `nifi-keycloak-client` Secret created by the shared `configure-stackable-oidc` Job. `NifiCluster.spec.clusterConfig.authentication` swaps from `simple-nifi-users` to `nifi-keycloak`. The old `simple-nifi-users` AuthenticationClass and `nifi-admin-credentials` Secret are deleted from the manifests directory. NiFi auto-redirects unauthenticated visits to OIDC; no `/login/keycloak`-style trigger path is needed.

**Tech Stack:** Stackable NiFi operator, NiFi 2.7.x OIDC, Stackable AuthenticationClass v1alpha1, Keycloak.

**Spec reference:** `docs/superpowers/specs/2026-04-28-stackable-stack-oidc-design.md` § "Per-product specifics → NiFi". This plan is the riskiest of the five — it removes the existing static auth path rather than adding a new one — which is why the spec recommends running it last.

**Depends on:** `2026-04-28-stack-oidc-prerequisites.md` (the `nifi` Keycloak client must exist).

## Task 1: Add NiFi AuthenticationClass + Secret to the shared Job

**Files:**
- Modify: `platform/manifests/configure-stackable-oidc/job.yaml`

- [ ] **Step 1: Insert the NiFi apply block**

In `platform/manifests/configure-stackable-oidc/job.yaml`, append after the most recently inserted product block (or after the placeholder echo if no other product plan has landed):

```sh
              echo "Applying NiFi AuthenticationClass + client Secret..."
              kubectl apply -f - <<EOF
              ---
              apiVersion: authentication.stackable.tech/v1alpha1
              kind: AuthenticationClass
              metadata:
                name: nifi-keycloak
              spec:
                provider:
                  oidc:
                    hostname: ${KEYCLOAK_HOSTNAME}
                    port: ${KEYCLOAK_PORT}
                    rootPath: /realms/stackable-demo
                    principalClaim: preferred_username
                    scopes:
                      - openid
                      - email
                      - profile
                    providerHint: Keycloak
              ---
              apiVersion: v1
              kind: Secret
              metadata:
                name: nifi-keycloak-client
                namespace: platform
              type: Opaque
              stringData:
                clientId: nifi
                clientSecret: nifi-secret
              EOF
```

- [ ] **Step 2: Verify YAML still parses**

Run: `python3 -c "import yaml; list(yaml.safe_load_all(open('platform/manifests/configure-stackable-oidc/job.yaml')))"`
Expected: no output.

- [ ] **Step 3: Commit**

```bash
git add platform/manifests/configure-stackable-oidc/job.yaml
git commit -m "nifi-oidc: extend configure-stackable-oidc with NiFi AuthenticationClass + Secret"
```

## Manual verification (after deploy)

NiFi is the riskiest of the five product migrations because it changes the auth path entirely rather than augmenting it. After the NiFi pod restarts on the new AuthenticationClass:

1. Open `https://<NODE_IP>:<nifi-nodeport>/` in incognito. Accept the self-signed cert.
2. NiFi auto-redirects to Keycloak.
3. Log in as `demo-admin`. Land on NiFi UI.
4. Top-right user identity shows `demo-admin`. Try opening Operate → Policies → Global access — should be allowed (initial admin can edit policies).
5. Log out, log in as `demo-user`. Land on NiFi UI. User shown as `demo-user`. Most policy-editing actions denied (no policies grant viewer rights yet).

Diagnostic if NiFi rejects the login:

- `kubectl -n platform logs nifi-node-default-0 -c nifi --tail=300 | grep -iE "oidc|oauth|claim|principal"`.
- `kubectl -n platform get authenticationclass nifi-keycloak -o yaml`.
- NiFi's own `/nifi-api/access/oidc/exchange` endpoint failing usually means the issuer URL doesn't match the JWT's `iss` claim. Confirm `<NODE_IP>:30900` is the value Keycloak emits.

Recovery if NiFi locks itself out:

- The initial admin identity is configured by the operator from the AuthenticationClass + the `preferred_username` claim of the first user with the `admin` realm role. If somehow the first user to register in NiFi isn't `demo-admin` (e.g. someone logs in as `demo-user` first), the NiFi authorizer files have to be reset. The Stackable operator regenerates them on PVC reset; the recovery is to delete NiFi's PVC and let the operator recreate. For a demo this is acceptable — there is no persistent flow data worth keeping yet.
